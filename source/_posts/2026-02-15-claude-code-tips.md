## 6 个核心技巧

### 1. 先用 Plan Mode，强制规划

在使用 Claude Code 之前，先进入计划模式：
- 明确任务目标
- 规划执行步骤
- 避免盲目开始

### 2. 创建并维护 CLAUDE.md

这是一个项目规范文档，应该包含：

| 内容 | 说明 |
|------|------|
| 项目规范 | 代码风格、命名规则 |
| 架构决策 | 技术选型、模块设计 |
| 常用命令 | 常用操作、快捷指令 |
| 过去犯过的坑 | 经验教训 |

**保持短小精悍，不断更新/修订。**

### 3. /compact + /compact 组合拳

**问题**：聊天历史太长 → token 爆炸

**解决方案**：
1. **/compact**：总结关键点
2. **/clear**：清空上下文
3. 只把**总结 + 必要文件**粘回去

### 4. 多用 Plan → Execute → Review 循环

| 阶段 | 操作 |
|------|------|
| **Plan** | 规划任务 |
| **Execute** | 执行计划 |
| **Review** | 审查结果 |

### 5. 拆解复杂任务

遇到复杂任务时：
- 先拆成 **sub-tasks**
- 每个小步都让 Claude **独立规划**
- 积小胜为大胜

### 6. 三个不要

| 不要 | 原因 |
|------|------|
| ❌ 一次性改太多文件 | 难以追踪 |
| ❌ 忽略 diff 预览 | 可能引入错误 |
| ❌ 让上下文无限膨胀 | token 爆炸、成本飙升 |

## 我的实践

### 1. CLAUDE.md 的价值

我一直维护着一个项目笔记，但看了这篇后意识到：
- 可以更系统化
- 加入代码规范和踩坑记录

### 2. /compact 的重要性

之前经常忘记 compact，导致：
- 上下文越来越长
- 回答质量下降
- 成本飙升

**以后要养成习惯**：长对话后先 compact 再继续。

### 3. 任务拆解

> "不要一次性让 Claude 改太多文件"

这个太对了！
- 每次改 1-2 个文件
- 及时 Review
- 确认无误后再继续

## 结论

AI 编程工具的使用是有**方法论**的：

1. **规划先行**：不要盲目开始
2. **保持精简**：CLAUDE.md + 定期 compact
3. **小步快跑**：拆解任务，及时 Review
4. **控制范围**：不要一次性改太多
